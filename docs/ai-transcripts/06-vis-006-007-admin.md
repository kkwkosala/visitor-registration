async getAllRequests(): Promise<{ data: VisitRequest[]; error: string | null }> {
  const { data, error } = await this.supabase
    .from('visit_requests')
    .select('*')

### Reactive Filter with Computed Signal
```typescript
protected filterStatus = signal<RequestStatus | 'all'>('all');

protected filteredRequests = computed(() => {
  const f = this.filterStatus();
  return f === 'all'
    ? this.allRequests()
    : this.allRequests().filter(r => r.status === f);
});
```

### Approve/Reject Modal with Optimistic Update
```typescript
protected async submitAction(): Promise<void> {
  const target = this.actionTarget();
  if (!target) return;

  this.submitting.set(true);
  const dto: UpdateRequestStatusDto = {
    status: this.actionNewStatus(),
    admin_comment: this.actionComment() || undefined,
  };

  const { error } = await this.requestSvc.updateRequestStatus(target.id, dto);

  if (error) {
    this.actionError.set(`Action failed: ${error}`);
  } else {
    // Optimistic: update in-memory list without refetching
    this.allRequests.update(list =>
      list.map(r =>
        r.id === target.id
          ? { ...r, status: dto.status, admin_comment: dto.admin_comment ?? null }
          : r
      )
    );
    this.closeAction();
  }
  this.submitting.set(false);
}
```

## Type Fix Identified by AI

**Problem:** `UpdateRequestStatusDto.status` was typed as `'approved' | 'rejected'` but assigned
a `RequestStatus` signal value (which includes `'pending'`):

```typescript
// Before — too narrow:
export interface UpdateRequestStatusDto {
  status: 'approved' | 'rejected';
}

// After — aligned with full union:
export interface UpdateRequestStatusDto {
  status: RequestStatus;
}
```

> "The DTO originally excluded 'pending' as a valid status. While semantically correct
> (admins shouldn't set pending), it caused a TypeScript assignment error with the computed
> signal. Broadened to `RequestStatus` and relied on UI controls to restrict what the admin
> can actually submit."

## AI Security Review

> "Admin identity is verified server-side via RLS:
> ```sql
> EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'admin')
> ```
> The client role badge is display-only. An attacker with a visitor JWT cannot call
> `updateRequestStatus()` and have it succeed — Supabase rejects it at the policy level."

> "Optimistic update does not re-fetch from the server. If the server update somehow fails
> after the optimistic patch (shouldn't happen with current error handling), the UI could
> show stale data. Acceptable for hackathon scope — a retry/reload button covers this."
