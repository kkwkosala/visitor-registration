# UI/UX Wireframes — Visitor Registration System

Text-based wireframes generated by AI during the UI/UX design phase.
These drove component decisions and interaction flows before any code was written.

---

## 1. Login Page (`/login`)

```
┌─────────────────────────────────────────┐
│                                         │
│         Visitor Registration            │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │  Email address                    │  │
│  │  [_______________________________]│  │
│  │                                   │  │
│  │  [    Send Magic Link    ]         │  │
│  └───────────────────────────────────┘  │
│                                         │
│  ✉️  Check your inbox — link sent       │  ← success state
│  ❌  Failed to send link. Try again.    │  ← error state
│  ⏳  Sending…                           │  ← loading state
│                                         │
└─────────────────────────────────────────┘

Decisions:
- Magic link only — no password field (ADR-002)
- Three exclusive states: idle | loading | success | error
- No "register" link — Supabase auto-creates account on first magic link
```

## 2. Auth Callback (`/auth/callback`)

```
┌─────────────────────────────────────────┐
│                                         │
│         ⏳ Signing you in…              │
│                                         │
│  [============================  ]  80%  │
│                                         │
└─────────────────────────────────────────┘

Decisions:
- Spinner shown during token exchange + profile load
- 8-second timeout fallback to /login
- Auto-redirects to role-appropriate dashboard (no user choice needed)
```

## 3. Navbar (shared, top of every page)

```
┌─────────────────────────────────────────────────────────────┐
│  🏢 Visitor Registration  │  [visitor] alice@example.com  [Sign out] │
└─────────────────────────────────────────────────────────────┘

Decisions:
- Role badge always visible (prevents confusion about current context)
- Email shows who is signed in
- No navigation links in navbar (routing handled by role-aware redirect on login)
```

## 5. Visit Request Form (`/visitor/requests/new` and `/visitor/requests/:id/edit`)

```
┌─────────────────────────────────────────────────────────────┐
│  New Visit Request                                          │
│  ─────────────────                                          │
│                                                             │
│  Visitor Name *                                             │
│  [_____________________________________________]            │
│  ❌ Visitor name is required                                │
│                                                             │
│  Email *                                                    │
│  [_____________________________________________]            │
│  ❌ Valid email required                                    │
│                                                             │
│  Purpose *                                                  │
│  [_____________________________________________]            │
│                                                             │
│  Visit Date *                                               │
│  [_____________________________________________]            │
│  ❌ Visit date must be today or in the future               │
│                                                             │
│  [Cancel]                    [Submit Request] ⏳            │
│                                                             │
│  ── Non-pending guard (edit mode only) ─────────────────   │
│  ⚠️  This request has been approved.                        │
│      Only pending requests can be edited.                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘

Decisions:
- Same component for create and edit (dual-mode via input() signal)
- minDateValidator prevents past visit dates
- Non-pending guard shows error instead of form (not silent redirect)
- markAllAsTouched() on submit to show all validation errors at once
```

## 6. Admin Dashboard (`/admin/dashboard`)

```
┌─────────────────────────────────────────────────────────────┐
│  🏢 Visitor Registration  │  [admin] admin@…  [Sign out]    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌────────┐  ┌─────────┐  ┌──────────┐  ┌──────────────┐  │
│  │   12   │  │    5    │  │    4     │  │      3       │  │
│  │ Total  │  │ Pending │  │ Approved │  │   Rejected   │  │
│  └────────┘  └─────────┘  └──────────┘  └──────────────┘  │
│                                                             │
│  Filter: [All] [Pending] [Approved] [Rejected]              │
│                                                             │
│  ┌───────┬──────────┬────────────┬──────────┬───────────┐  │
│  │Visitor│  Email   │Visit Date  │ Status   │ Actions   │  │
│  ├───────┼──────────┼────────────┼──────────┼───────────┤  │
│  │ Bob   │bob@…     │ 2099-12-01 │[pending] │[✓][✗]     │  │
│  │ Carol │carol@…   │ 2099-12-05 │[approved]│    —      │  │
│  └───────┴──────────┴────────────┴──────────┴───────────┘  │
│                                                             │
└─────────────────────────────────────────────────────────────┘

Decisions:
- Stats derived from same signal as table (no extra DB call)
- ✓ = Approve, ✗ = Reject — only shown for pending
- Filter buttons use signal — no page reload
```

## 7. Approve/Reject Modal (overlay on admin dashboard)

```
┌─────────────────────────────────────────┐
│  Approve Visit Request                  │
│  ─────────────────────                  │
│  Visitor: Bob Smith                     │
│  Date: December 1, 2099                 │
│                                         │
│  Comment (optional)                     │
│  ┌─────────────────────────────────┐    │
│  │ Welcome! Please bring your ID.  │    │
│  └─────────────────────────────────┘    │
│                                         │
│  ❌ Action failed: network error        │  ← error state
│                                         │
│  [Cancel]          [Confirm Approve] ⏳ │
│                                         │
└─────────────────────────────────────────┘

Decisions:
- Inline modal (no separate route) — faster for admin workflow
- Comment is optional — admins can approve/reject silently
- Optimistic UI update on confirm (no page reload)
- Cancel button is the accessible close mechanism (no backdrop click)
```
